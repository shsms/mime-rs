* Vision

~mime-rs~ (reads "mimers") is a headless, *stateful* text-editing engine that an AI
agent drives by submitting small programs. It is the spiritual heir to shsms's
~mime~ (a ChaiScript-scripted buffer editor), rebuilt in Rust on top of [[file:../tulisp][tulisp]]
(an Emacs-Lisp dialect), and re-aimed squarely at agentic use.

The one-line thesis: *today's agent edit tools are stateless and
content-addressed ("replace this exact string"); they cannot express
programmatic, structure-aware, bulk edits, so agents fall back to ad-hoc shell
or Python and reinvent buffer operations badly.* ~mime-rs~ gives the agent a
third option between "brittle single-string edit" and "arbitrary code
execution": a bounded, auditable, *transactional* editing language with warm
buffers that persist across calls and checkpoints it can rewind to.

Pitch in one breath: *run an edit program against warm buffers, see the diff,
keep it or roll back to any checkpoint — over MCP, sandboxed, with no
re-reading of files.*

* Why this, why now

** The gap in current AI editing tools
- /Stateless/: every edit re-reads the file and reproduces exact strings. State
  ("where am I", "what's selected") cannot be carried between operations.
- /Content-addressed only/: "find old_string → new_string" cannot say "for every
  match, do X within its enclosing region", "after the next =</p>=", "between
  point and mark", or "only within this narrowed section".
- /No transaction/: a wrong edit is not atomically revertable. There is no cheap
  "try this whole program; if it's wrong, undo all of it".
- /No structured feedback/: tools return prose or a whole file, not a diff +
  machine-readable signals an agent can branch on.
- /Stale-read races/: the agent's copy of the file drifts from disk; cf. the
  "File has been modified since read" failure mode.

** Lessons banked from `mime` (the predecessor)
Building the ulysses-annotated pipeline on ~mime~ taught us concretely what to
keep and what to fix. ~mime-rs~ exists to keep the wins and kill the cuts.

| From mime                          | Verdict | mime-rs response                                              |
|------------------------------------+---------+--------------------------------------------------------------|
| buffers + multiple cursors         | keep    | first-class buffers + markers (positions that survive edits) |
| ~narrow_to_region~ (bound a slice) | keep    | ~narrow-to-region~ + ~save-restriction~                      |
| ~find_fuzzy~ (ws/case-insensitive) | keep    | ~search-fuzzy~ with configurable normalization               |
| batch model (write a program, run) | keep    | batch is the *primary* mode; daemon adds warm state          |
| ~replace~ is exact-only            | cut     | ~replace-regexp~ with capture groups, count, replace-in-region |
| ~replace(regex(...),...)~ crashes  | cut     | regex is a first-class arg, not a dispatch accident          |
| ~"\xa0"~ literal panics (bad UTF-8) | cut     | never panic on bytes; UTF-8-safe everywhere                  |
| output via ~print~, scrape stdout  | cut     | structured ~(report k v)~ + auto unified diffs               |
| single-shot, cold every run        | cut     | daemon keeps buffers + defs warm across programs             |
| no undo / no dry-run               | cut     | checkpoint tree + transactions + ~rehearse~ (dry-run)        |
| no audit trail                     | cut     | every program + diff logged and replayable                   |

** Why tulisp
- Emacs Lisp + buffers/markers/narrowing is the most battle-tested text-editing
  vocabulary in existence; we adopt its model rather than invent one.
- We *own* the interpreter: editor primitives are plain Rust closures registered
  via ~ctx.defun("name", |..| -> ..)~ (identical to how tulisp ships its own
  builtins, e.g. ~make-hash-table~). Buffers, markers, regions, and checkpoints
  are exposed as opaque tulisp objects (same pattern as tulisp's ~HashTable~).
- tulisp already has a *bytecode compiler* (=src/bytecode=), so hot edit programs
  can be compiled once and replayed cheaply against warm buffers.
- Dogfoods tulisp on a demanding, real workload — which matures tulisp the same
  way the games push does.

* Design principles
1. *Batch-first.* The unit of work is a /program/, not a keystroke. One
   round-trip submits a whole edit program; we never make the agent issue
   cursor-moves one HTTP call at a time. (Round-trips are the thing that kills
   interactive editor protocols for LLMs.)
2. *Warm state.* A long-lived daemon owns buffers, kill-ring, registers, agent
   helper definitions, and the checkpoint tree. Re-reading and re-deriving
   context between programs is the cost we are deleting.
3. *Transactional & reversible.* Every program is atomic by default; any state is
   recoverable from a checkpoint. "Try it, look, keep or rewind" is the core loop.
4. *Structured I/O.* Return diffs and machine-readable reports, not prose or whole
   files. Be token-frugal: diffs and summaries by default, full text on demand.
5. *Auditable & sandboxed.* A bounded command surface with no shell/network/arbitrary
   FS. Granting "mime-rs edit" must be safe in a way that "grant bash" is not.
6. *UTF-8-safe & panic-free.* Malformed input, missing files, and odd bytes are
   errors, never crashes.
7. *Deliberate Emacs subset.* We are re-implementing a chosen /slice/ of Emacs's
   buffer model. The slice is explicit (see §6); we resist becoming Emacs.

** Components
- *Editor core* — buffers (piece trees over mmapped backing), point, mark, markers, regions, narrowing;
  UTF-8-safe; the actual text data structure and its operations.
- *tulisp editor API* — Rust closures registered on a ~TulispContext~ that expose
  the editor core to programs (§6). Buffers/markers are opaque tulisp objects.
- *Session manager* — maps a session id to a /workspace/ (a set of buffers + kill
  ring + registers), a tulisp context (with warm agent definitions), a
  checkpoint tree, and an audit log. Handles isolation and concurrency.
- *Checkpoint/transaction layer* — copy-on-write snapshots of the whole workspace;
  the undo/redo/branch/rewind machinery (§7).
- *Transport* — the daemon ~mimed~ exposes one control API (HTTP/JSON over a unix
  socket). Three clients speak it: ~mimectl~ (the CLI — what you use instead of
  hand-rolling HTTP), an MCP server (the agent surface), and raw HTTP. All wrap the
  same engine; a ~mimectl~ one-shot can also run the engine embedded, no daemon.
- *Audit/journal* — append-only record of every submitted program, its diff, and
  its resulting checkpoint, for replay and trust.

* The editor core (data model)

Files can be several GB — larger than RAM — so the buffer must *not* hold all the
text. We use a *persistent piece tree over immutable, memory-mapped backing*: VS
Code's piece tree, made copy-on-write so it doubles as the checkpoint mechanism.

- *Backing stores* (immutable, shared by every snapshot):
  - /original/ — the file, ~mmap~ed read-only; the OS pages it in/out, never fully
    resident. Behind a trait, so an explicit paged-LRU reader (bounded memory, no
    SIGBUS) can replace ~mmap~.
  - /add buffer/ — append-only, holds *only* inserted text; spills to a temp file if
    it grows large.
- *Buffer* := a *persistent measured B-tree of pieces* + a name + optional file path
  + per-buffer /point/, /mark/, /markers/, /narrowing/ (min,max), dirty and read-only
  flags. A /piece/ is ~(source, start, len)~ — a reference into a backing store, never
  owned text. An unedited GB file is one piece; an edit splits pieces and points a new
  one at the add buffer; original bytes never move.
- *Tree*: ~Rc~ nodes, path-copying on edit (O(log n) nodes copied); each node carries
  a monoid /summary/ (bytes, chars, lines, + a hook for a marker index) so we seek by
  byte/char/line in O(log n). Same measured persistent B-tree as a rope — only the
  leaf payload changes from owned string to piece reference, so the spine can be our
  own or a vendored ~sum_tree~.
- *Snapshots are ~KB, even for GB files*: a checkpoint clones only the piece-tree
  spine; both backing stores are shared with zero copying. Buffer memory ≈ edits + the
  OS page cache (evictable).
- *Coordinates*: char positions, 1-based, Emacs-style (~point-min~ = 1); byte offsets
  never leak; line/column accessors provided; no op splits a UTF-8 scalar. char/line
  indices over a byte-addressed original need a scan to find boundaries — built lazily
  / in the background / from a persisted sidecar, never blocking the first edit.
- *Markers / anchors*: an anchor = ~(backing-id, byte-offset, bias)~ into an immutable
  store, so it is stable across *every* snapshot for free; resolving it to a line/col
  needs only the target snapshot's tree. (Offset-shifted-on-edit markers also work
  within one timeline; anchors are what survive the checkpoint *tree*.)
- *Non-UTF-8* input (binary, odd logs): rejected at open initially; an explicit byte
  mode can come later.
- *Workspace*: the set of buffers in a session + the shared kill-ring + registers.
  A *checkpoint snapshots the whole workspace*, not a single buffer.

~kill-ring~ access; ~set-register~/~get-register~ to stash text, positions, or

** Observability (return data to the agent)
| ~(report KEY VALUE)~ | attach a machine-readable signal to the result (e.g. counts) |
| ~(message FMT ...)~  | human-readable log line                                      |
| ~(echo-region B E)~  | ask the result to include this snippet (on-demand context)   |

Tree-sitter-backed: ~(treesit-search QUERY)~, ~(node-at-point)~, ~(narrow-to-defun)~,

This is where ~mime-rs~ becomes a serious *code* editor, not just text.

** Warm definitions
A session may preload a tulisp /library/ (agent-defined helper ~defun~s),
evaluated once and persisted across programs. So the agent teaches the editor a
vocabulary ("join-adjacent-links", "wrap-in-tag") and reuses it cheaply.

* Checkpoints, transactions, and time travel   :headline:

This is the feature the user asked for first and the one agents need most.

** Model
- A *checkpoint* is a snapshot of the *entire workspace*: every buffer's text +
  point/mark/markers/narrowing, plus kill-ring and registers.
- Checkpoints form a *tree* (not a stack): restoring an old checkpoint and editing
  forward creates a branch (undo-tree semantics), so the agent never loses a
  state by exploring an alternative.
- Snapshots are *cheap* even for multi-GB files: the immutable backing stores
  (mmapped original + append-only add buffer) are shared, so a checkpoint copies only
  the piece-tree spine (~KB) — O(1) in *file* size, paid lazily on later edits.

** Operations
| ~(checkpoint "label")~        | create a named checkpoint, return its id                 |
| ~(restore-checkpoint id)~     | rewind the whole workspace to it (current branch updates) |
| ~(list-checkpoints)~          | the tree: id, label, parent, time, buffers touched       |
| ~(checkpoint-diff A B)~       | unified diff of the workspace between two checkpoints     |
| ~(current-checkpoint)~        | where we are                                             |
| ~(forget-checkpoint id)~      | GC an unwanted branch                                    |

** Automatic transaction per program
Every submitted program is implicitly wrapped:
1. auto-checkpoint the workspace *before* the program runs;
2. evaluate the program;
3. on *success* → keep the changes, the pre-checkpoint remains as the rewind point;
   on *tulisp error / explicit ~(rollback)~* → atomically restore the
   pre-checkpoint; the result reports ~rolled_back_to~. No partial mutation leaks.
~(with-transaction ...)~ exposes the same all-or-nothing guarantee at sub-program
granularity for risky inner steps.

** Dry-run / rehearse
~rehearse~ mode (CLI ~--rehearse~, MCP ~dry_run: true~) runs the program, computes the
resulting diffs and reports, then *always rolls back* and returns the diff
without committing. "Show me what this program /would/ do." This would have saved
dozens of full epub rebuilds in the ulysses work.

** Why this matters for agents
Agents are wrong often. Cheap, named, *branching* rewind turns "I made a mistake,
now reconstruct the file" into ~(restore-checkpoint "before-refactor")~. Combined
with ~rehearse~, the agent's loop becomes: rehearse → inspect diff → commit or
adjust → checkpoint. Mistakes cost one call, not a salvage operation.

** Token discipline
- Default to *diffs and summaries*, never whole buffers.
- Cap diff size; when exceeded, return a stat summary + a handle to fetch ranges.
- The agent pulls full text only via explicit ~read_region~ / ~(echo-region ...)~.
- Positions, match data, and counts are returned numerically so the agent can act
  without re-reading.

* Daemon, sessions, and concurrency

- *Daemon* (~mimed~ — "mimed", as in mime + d): long-lived, owns all sessions and
  serves the control API. Clients reach it via ~mimectl~, MCP, or raw HTTP. A
  ~mimectl~ one-shot can also run the engine in-process without a daemon.
- *Session* = isolated workspace + tulisp context + checkpoint tree + audit log.
  Mapped 1:1 to an agent conversation by convention. Created explicitly; idle-
  evicted on a TTL; buffers in a session never leak to another.
- *Concurrency*: programs within a session are serialized (a session is a single
  writer — simplest correct model; edits are inherently ordered). Different
  sessions run in parallel. Per-buffer write locks guard shared on-disk files
  touched by multiple sessions.
- *Durability*: buffers are in-memory; text reaches disk only via ~save-buffer~ /
  ~write-file~. Optional WAL journaling of programs lets a crashed daemon replay
  unsaved sessions. Checkpoints are in-memory by default, with an opt-in to
  persist the tree to disk for cross-restart time travel.
- *Lifecycle/limits*: max buffers, max total bytes, max checkpoints per session,
  eviction policy (named checkpoints pinned; anonymous GC'd beyond N).

* Transport: one API (~mimed~), three clients

The daemon ~mimed~ serves a single control API; ~mimectl~, MCP, and raw HTTP are
three front-ends to it. You should rarely hand-write HTTP — reach for ~mimectl~.

** mimectl (the CLI — instead of direct HTTP)
A thin, scriptable client over the daemon API (systemctl/kubectl-style verbs):
| ~mimectl run PROG.tl [--file F] [--rehearse]~       | submit a program (dry-run with --rehearse) |
| ~mimectl open FILE [--read-only]~                   | attach a file to the session            |
| ~mimectl ls~ / ~mimectl status~                     | buffers, dirty flags, current checkpoint |
| ~mimectl search BUF PAT [--regex\vert--fuzzy]~      | run a search, print matches             |
| ~mimectl checkpoint [LABEL]~ / ~mimectl checkpoints~ | create / list the checkpoint tree       |
| ~mimectl restore ID~ / ~mimectl diff A B~           | rewind / show a checkpoint diff          |
| ~mimectl save BUF~ / ~mimectl write BUF PATH~       | persist to disk                          |
| ~mimectl repl~                                      | interactive tulisp against the session   |
~mimectl~ resolves its session from ~$MIME_SESSION~ or ~--session~; with ~--local~ it
runs the engine embedded (no ~mimed~ needed) for one-shots.

** MCP tools (the agent-facing surface)
High-level conveniences, with ~run_program~ as the power-user escape hatch:
| ~open_file(path, read_only?)~                       | attach a file                     |
| ~run_program(program, session?, dry_run?)~          | eval a tulisp edit program        |
| ~search(buffer, pattern, mode)~                     | exact/regex/fuzzy, returns matches |
| ~read_region(buffer, start, end \vert around_point)~ | pull text on demand               |
| ~checkpoint(label)~ / ~list_checkpoints~ / ~restore_checkpoint(id)~ / ~diff(a,b)~ | time travel |
| ~save_buffer(buffer)~ / ~write_file(buffer, path)~  | persist                           |
| ~session_status()~                                  | buffers, dirty flags, current checkpoint |
MCP gives tool schemas, permissioning, and discovery for free — the right
integration surface for agents. HTTP/JSON mirrors these for non-MCP clients.

** HTTP/JSON (the wire ~mimectl~ and MCP ride on)
The raw control API, served by ~mimed~ over a unix socket (TCP optional):
~POST /sessions~, ~POST /sessions/:id/run~, ~GET /sessions/:id/buffers~,
~POST /sessions/:id/checkpoints~, ~POST /sessions/:id/restore~,
~GET /sessions/:id/diff?a=..&b=..~. Sync by default; long programs may return a job
id + ~GET /jobs/:id~. ~mimectl~ is just a friendly skin over these.

* Safety, sandboxing, permissioning

- *Bounded vocabulary*: tulisp programs see only the editor builtins + pure lisp.
  No shell, no process spawn, no network, no arbitrary FS. ~open-file~/~write-file~
  are restricted to configured workspace roots (path allowlist, symlink-checked).
- *Read-only buffers*: reference material can be attached unwritable.
- *Resource limits*: per-program wall-clock timeout, memory cap, max buffer size,
  max output size; eval is interruptible (cooperative cancellation in the
  bytecode interpreter).
- *Determinism*: no ambient time/randomness except via explicit, logged builtins.
- *Auditability*: every program source + its diff + resulting checkpoint is
  journaled. You can answer "what did the agent change, and why" and *replay* it.
- *The pitch*: this is why "let the agent run mime-rs" is safe where "let the agent
  run bash" is not — same editing power, none of the blast radius, full rewind.

* Performance

- *The file is never fully resident.* Piece tree over mmapped (or paged-LRU)
  backing: editing or snapshotting a multi-GB file touches O(log n) pieces and the
  OS page cache, not the bytes.
- *Checkpoints are O(spine), ~KB*, at any file size — immutable backing is shared
  across all snapshots; only the piece-tree path is copied.
- *Indexing is the costly op, not editing.* char/line summaries need one scan of the
  original to find boundaries — done lazily / in the background / from a persisted
  sidecar, so it never blocks the first edit.
- *Search* streams a cursor across pieces (a ~regex-automata~ incremental DFA, so
  matches can cross piece/backing boundaries); the common all-original case scans the
  ~mmap~ directly (~memchr~ / ~regex::bytes~) and maps offsets back.
- *Saving is O(file size)* by nature — stream every piece to a new file + atomic
  rename; a tiny edit to a huge file still rewrites it. Budgeted, not hidden.
- No full re-serialization per op (mime re-walked a 43 MB buffer repeatedly); track
  dirty spans, diff only changed ranges, and *never materialize the whole buffer* —
  the API stays ranged/streamed (mandatory at GB scale, not just token-thrift).
- Compile hot programs via tulisp's bytecode path; cache compiled programs per
  session keyed by source hash.
- *Targets*: open + edit a multi-GB file at flat memory (≈ edits + page cache);
  checkpoint create/restore ~µs–ms at any size; typical agent program sub-second; the
  ulysses-annotated transform (≈43 MB) as a standing perf+correctness benchmark.

* Testing & validation
- *Unit*: each builtin, incl. UTF-8 edge cases (NBSP, combining marks, lone
  surrogates rejected) — the ~\xa0~ crash becomes a permanent regression test.
- *Property*: ∀ program P and checkpoint C, ~restore(C)~ after P == state at C;
  ~undo~∘~edit~ == identity; markers track edits correctly.
- *Golden*: (program + input buffer) ⇒ expected unified diff.
- *Fuzz*: random edit programs never panic and never corrupt UTF-8.
- *Integration*: the ulysses port (above); concurrent-session isolation;
  daemon crash/replay from the WAL.
- *Bench*: the 43 MB transform; open/edit a multi-GB file at flat memory; checkpoint
  create/restore at any size.

* Open questions & risks (decide deliberately)
1. *Why not ~emacsclient~ + elisp?* It already has buffers/markers/narrowing/regexp
   and a server. Our answer: lean single-binary embeddability, a shaped/bounded
   primitive set, JSON-native results, transactions-for-agents, and a sandbox we
   control. Valid — but we are choosing to reimplement a /slice/ of Emacs; name
   the slice and stop there.
2. *Why not sandboxed Python + a buffer lib?* The differentiators must be: warm
   cross-call state, an auditable bounded surface, and first-class transactions.
   If those don't feel decisive, the niche is thin — pressure-test this early.
3. *tulisp maturity/perf*: is the interpreter (even bytecode-compiled) fast enough
   over multi-GB buffers? This is a forcing function for tulisp; budget for it.
4. *Checkpoint memory & persistence*: GC policy for branches; do checkpoints
   survive a daemon restart? (WAL replay vs. persisted tree.)
5. *Multi-session edits to the same on-disk file*: conflict/lock policy; last-
   writer-wins vs. detect-and-refuse.
6. *Coordinate model*: confirmed char-based, 1-based, byte-safe — but line-oriented
   tools may want line/col first-class too.
7. *Diff granularity vs. token budget*: where to truncate, and the fetch-on-demand
   contract.
8. *Scope creep*: the gravitational pull toward "reimplement Emacs". Guard with the
   principle in §3.7 and the explicit milestone exits.

* Appendix A — end-to-end example

A program that, within one chapter slice, joins adjacent same-note links, counts
them, and checkpoints — run in ~rehearse~ first to preview the diff.

#+BEGIN_SRC tulisp
(with-current-buffer (open-file "index.php-cyclops.htm")
  (let ((b (point-min)) (e (point-max)) (joined 0))
    (save-restriction
      (narrow-to-region
        (search-forward "[ 12 ]") (search-forward "[ 13 ]"))
      (goto-char (point-min))
      ;; merge </a><a ...same href...> runs
      (while (re-search-forward "</a><a href=\"\\([^\"]+\\)\"[^>]*>" nil)
        (when (same-note-p (match-string 1))      ; a warm helper def
          (replace-match "")
          (setq joined (1+ joined)))))
    (report "joined" joined)
    (checkpoint "cyclops-links-joined")))
#+END_SRC

The result returns the unified diff for the buffer, ~"reports": {"joined": N}~, and
the new checkpoint id — committed only if not run as ~rehearse~.

* Appendix B — naming & lineage
- Project / engine crate: ~mime-rs~ (reads "mimers"; the ~-rs~ Rust suffix completes
  the word).
- Daemon binary: ~mimed~ ("mimed" — mime + d, and the past tense fits: it has
  already performed your edits).
- CLI binary: ~mimectl~ (systemctl/kubectl convention — the control client you use
  instead of raw HTTP).
- Lineage: shsms's ~mime~ → ~mime-rs~. Same buffers-and-cursors soul; new powers
  (transactions, structured I/O, MCP) and none of the cuts.
- Tagline candidate: /"an amanuensis for your agents — it takes dictation, keeps
  every draft, and never loses your place."/
