** TODO Paged reader holds an open fd per buffer [#C]
DECIDED (2026-06): a drifted paged buffer must NOT re-open the path
mid-life — swapping inodes under a live piece tree mixes old and new
bytes, and the old inode IS the correctness anchor for a modified
buffer's edits. The exposure is mitigated at the session layer instead:
auto-revert re-opens clean drifted buffers on next use, the warm-session
cap (clean-LRU eviction) bounds held fds, and ~close_session~ releases
deliberately — together these also bound the ~.nfsXXXX~ silly-rename
lingering on NFS (now gitignored). Remaining idea, if fd pressure ever
shows up in practice: a re-open-on-demand mode for CLEAN buffers only.

~build_tool_schemas~ (mcp.rs), the separate ~help~ prose (help.rs), and (soon) a
third copy in ~instructions~. Collapse them: ONE registry entry per tool, every
doc surface a PROJECTION of it — nothing hand-synced.

Per-tool ~ToolDoc~: ~name~ · ~summary~ (one line) · ~description~ (full) ·
~input schema~ (properties + required) · ~annotations~ (read_only / destructive /
idempotent / open_world) · ~examples~ (call + note) · ~topics~ (which ~help~

session / introspection) · ~see_also~. A single ~catalogue() -> Vec<ToolDoc>~.

  not a mix. Merging buys nothing there.

- DROP ~wrap~ from the action set. ~replace~ / ~delete~ / ~insert-before~ /

The automated composition of two things mime already has: git_blame
{worktree: true} maps each uncommitted hunk to the commit that owns its
lines, and git_fixup {hunks} folds selected hunks into a target. absorb
runs the loop: blame every dirty hunk, group by owning commit, fold each
group — one call, no target names. Hunks whose owner is ambiguous (outside
the branch, or split ownership) stay in the worktree and are reported.
rehearse:true previews the grouping. Modeled on magit-commit-absorb /
git-absorb.

Field evidence [2026-07-15]: one review-fix session hand-rolled this
loop FOUR times (dump -U0 diff, split hunks by blame target with a
python script, apply+commit per group, autosquash) — the single largest
avoidable tool-call sink of the session. Two notes from that use: (1) a
grouped blame view (git_blame {worktree: true, group_by: "commit"} →
{commit: [hunks]}) is the natural input and useful on its own; (2) some
hunks need routing by NAME, not blame (a renamed fn's call sites belong
with the rename commit even when blame says otherwise), so ambiguous or
overridden routing must stay expressible — absorb the clear ones, leave
the rest in the worktree, report.

Recurred [2026-07-15, formula-explorer gate]: the same
distribute-hunks-to-owning-commits loop, hand-rolled with an even
cruder tool — revert file, re-apply hunk A by editor, git commit
--fixup=X, re-apply hunk B, fixup Y — four times in one session, plus
one mis-attributed hunk that had to be re-reverted mid-dance. Every
occurrence confirms the by-name routing note: review fixes cluster by
COMMIT TOPIC, not by blame line.
