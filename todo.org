** TODO Paged reader holds an open fd per buffer [#C]
DECIDED (2026-06): a drifted paged buffer must NOT re-open the path
mid-life — swapping inodes under a live piece tree mixes old and new
bytes, and the old inode IS the correctness anchor for a modified

shows up in practice: a re-open-on-demand mode for CLEAN buffers only.

*** Centerpiece: a self-documenting tool registry (single source of truth)

Today three things describe the tools and can drift: the hand-built schemas in
~build_tool_schemas~ (mcp.rs), the separate ~help~ prose (help.rs), and (soon) a
third copy in ~instructions~. Collapse them: ONE registry entry per tool, every
doc surface a PROJECTION of it — nothing hand-synced.

Per-tool ~ToolDoc~: ~name~ · ~summary~ (one line) · ~description~ (full) ·
~input schema~ (properties + required) · ~annotations~ (read_only / destructive /
idempotent / open_world) · ~examples~ (call + note) · ~topics~ (which ~help~
briefs it belongs to) · ~category~ (editing / structural / conflicts / git /
session / introspection) · ~see_also~. A single ~catalogue() -> Vec<ToolDoc>~.

Projections (all derived):
- ~tools/list~ → {name, description, inputSchema, annotations} per entry.
- ~initialize.instructions~ → a curated preamble (the engine model: warm
  sessions + path auto-open + ~save:true~; rehearse → run → save; ~undo_last~;
  ~expect_unique~; the tier/safety note) + an AUTO-generated compact index
  grouped by category (~name — summary~) + "call ~help {topic}~ for briefs". Lean
  (the client injects it into the model's context — keep to ~a screen, push
  detail to help/examples).
- ~help {topic}~ → curated topic prose (regex/treesit DIALECT briefs stay
  hand-written) + the registry entries tagged with that topic (summary +
  examples). Add a ~help {tool: "name"}~ form → that entry's full description +
  examples + annotations.
- ~describe-mcp~ / docs/mcp-tools.md → full catalogue from the registry (already
  generated; now also carries annotations + examples). ~validate_args~ keeps
  reading the tools/list projection, so it can't drift from what's accepted.

*** TODO git: load the git-tool schemas on demand, not always-on :mcp:
The real cost of "git is ~a third of the surface" is the ALWAYS-LOADED schema
weight, not the tool count itself — so cut what's always loaded, don't overload
the tools. The rebase/cherry-pick/conflict flow is rarely touched in a typical
editing session, yet its 9 schemas sit in every client's context from the first
turn. Load them on demand instead: keep the editing tools always-on, and surface
the ~git_*~ schemas only when a repo operation is in play (a deferred/lazy tool
set the client fetches by name — the pattern this very harness uses for its own
deferred tools). Cuts the always-loaded surface with ZERO loss of per-call
clarity: every git tool stays one-intent, flat-schema, git-named.

REJECTED alternative (considered 2026-06-29): merging the read verbs into
~git_history {query: log|show|status}~ and folding ~git_skip~ into ~git_continue
{skip?: true}~. Against, on agent-ergonomics grounds:
- The "one honest annotation" win is illusory for the read merge — ~git_log~ /
  ~git_show~ / ~git_status~ are ALREADY all read-only; three honest annotations,
  not a mix. Merging buys nothing there.
- It trades flat, single-purpose schemas for a DEPENDENT-parameter one (~range~
  only for log, ~commit~ required only for show) that JSON Schema can't express
  cleanly → prose + runtime validation → more wrong-call round-trips, worst for
  the weak models this surface targets.
- ~git_log/show/status~ are names the model already learned from git; a
  ~git_history {query}~ meta-name + enum is strictly more to get right.
- Folding skip into continue re-merges OPPOSITE consequence-classes under one
  boolean (continue commits the resolution = append; skip hard-resets, discarding
  the merge residue = destructive) — exactly the read+write / reversible+destructive
  mix that killed the polymorphic-checkpoint idea (principle 3 above). One
  annotation can't honestly gate both.
Net: the merge optimizes tool COUNT (a proxy) at the expense of per-call CLARITY
(the real driver of agent accuracy). Deferred loading gets the context saving
without that cost.

Assessment (2026-06-29): right instinct, but as written the proposal
over-scopes and under-specifies. Build the core, with these changes:
- DROP ~wrap~ from the action set. ~replace~ / ~delete~ / ~insert-before~ /
  ~insert-after~ each map cleanly to {one text, one position}; ~wrap~ does NOT.
  The existing primitive is ~(treesit-wrap-node NODE BEFORE AFTER)~
  (builtins.rs:2404) — it encloses a node by inserting on BOTH sides (expr →
  ~Some(~ … ~)~), so it needs TWO strings, not the proposal's single ~text~.
  Decorators / Rust ~#[...]~ are just the degenerate before-only case — i.e.
  plain ~insert-before~ — so the common defun-level want is already covered;
  true wrapping is mostly a sub-defun/expr op, fine to leave in ~run_program~.
- NAME DISAMBIGUATION is the make-or-break and the proposal is silent on it.
  Real code repeats names (two ~impl~ blocks each with ~new~, free fn vs
  method, trait impls, nested fns). An ambiguous ~name~ must be an ERROR that
  lists candidates (kind + enclosing scope), like ~replace_text~'s
  ~expect_unique~ — never a silent first-match. Probably needs a path-ish
  selector (~Foo::new~, or kind+name).
- "80% case" OVERSELLS it. The most common edit is SUB-defun ("change these
  three lines in ~foo~"), which stays ~replace_text~. ~edit_defun~ owns
  WHOLE-UNIT ops — a meaningful minority. Honest framing: "removes hand-written
  treesit lisp for whole-unit edits."
- Where it genuinely beats the alternatives: ~replace~/~delete~ of a named node
  needs no reproduction of the old body (literal ~replace_text~ does). Note the
  file's own "grow power via params" principle would argue for params on
  ~insert_text~/~replace_text~ (the latter already has ~scope: {defun}~) instead
  of a new tool — but a single named ~edit_defun~ wins HERE because it
  CONSOLIDATES one intent ("operate on a code unit by name") rather than
  scattering it across three tools' optional params; that's the opposite of the
  git-merge case, where a param overloaded one tool. Consistent, not special.
- DEPENDENCY: correctness rides on per-language defun spans, and Python
  ~decorated_definition~ spans are currently wrong (decorators excluded — see
  "More languages" above). Fix that first, or scope ~edit_defun~ to languages
  with correct spans.
- PRIORITY: the ~help {list: "operations"}~ companion is the higher-leverage
  half — structured discoverability of the whole ~run_program~ vocabulary (fed
  by the tulisp obarray-introspection wishlist item) is the SYSTEMIC cure for
  "power hidden behind run_program"; ~edit_defun~ is one curated shortcut. Do
  the help-list first.
