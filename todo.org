* Engine

** TODO Shrink upd-epub-to-jp.tl's packed error message
Failure JSONs now carry the failed run's ~reports~/~log~ across all three
front-ends, so the script can go back to a plain ~(error ...)~ with the
diagnostics living in reports/log again.

** TODO ulysses scripts: drop the set-buffer-home workaround
The closing report now tracks the primary buffer, so add-anno /

drop their end-of-run ~set-buffer~ dance.

** TODO Streaming search [#C]
DONE so far: literal ~search_forward~ streams the range a chunk at a time
(needle-1 overlap), stopping at the first hit; ~re_search_forward~ and
~looking_at~ are ADAPTIVELY WINDOWED — the window grows from a small start and
settles as soon as the match ends inside it, so a hit near point never copies
the document tail (was point→EOF, unconditionally for ~looking_at~). Together
these retire the conflict builtins' full-buffer ~scan~ cost.
Remaining: a resumable streaming DFA (~regex-automata~) to close the residual —
a regex MISS still grows the window to the bound (the engine must see every
byte to say "no"); streaming ~search_backward~ (still materializes [bound,
point)); and global-replace a multi-GB file across cores. Also: ~find_forward~
matches with a naive ~windows().position()~ — swap in ~memchr::memmem~ (SIMD
two-way) for long/common-prefix needles.
/Exit: global-replace a multi-GB file across cores; ulysses bench beats
legacy ~mime~./

** TODO Paged reader: fuse the open scans, coarsen the drift stat [#C]
Two perf follow-ups from the paged-reader landing, both scaling with file
size. (1) ~Quire::open~ reads the whole file TWICE — ~is_valid_utf8~ then
~count_chars_lines~ — where mmap+~from_utf8~ was one pass; fuse them into a
single page-by-page scan that validates UTF-8 AND tallies chars/lines (carry
an incomplete trailing char once). Also restore the parallel fan-out for the
paged count (it's owned-only now; threads can ~read_at~ disjoint ranges).
(2) The fresh-read drift check does a ~stat~-by-path per page MISS until it
latches — ~16k stats per GB on a cold scan of a clean file. Check once per
read op (or every K pages), not per 64 KiB page.

** TODO Paged reader holds an open fd per buffer — fd use + rename reads [#C]
Unlike the old mmap (which dropped the ~File~ after ~Mmap::map~), ~PagedFile~
keeps ~file: std::fs::File~ open for the buffer's whole life. Two consequences:
(1) one held fd per warm buffer — many warm daemon/MCP sessions = many fds;
worth a soft cap or a re-open-on-demand mode. (2) Reads go through the held
fd, so after an external RENAME-replace the buffer keeps serving the OLD
(now-unlinked but stable) inode's bytes while drift IS flagged via the
stat-by-path; on NFS the unlinked inode is silly-renamed to ~.nfsXXXX~ and
lingers until the session drops (visible when dogfooding mime on its own
NFS-hosted source). Decide whether a drifted paged buffer should re-open the
path on the next read (pick up the new inode) rather than wait for revert.

** TODO Dogfooding friction: MCP "insert relative to a defun/node" [#C]
From driving mime to edit mime-rs itself. DONE: the unsaved state is now
legible — ~session_status~ carries an ~unsaved~ flag per session and
~run_program~/~replace_text~/~insert_text~ flag/note an edit that lives only in
the warm buffer (file-backed + modified), so a forgotten ~save~ can't vanish
silently. Remaining: appending a large NEW block has no anchor —
~treesit-insert-sibling~ covers it in lisp, but an MCP-level "insert relative to
a defun/node" convenience (or a ~save~-by-default flag) would close the common
"add a function after X" case without hand-writing a ~run_program~. Possible
further step: a warm buffer with unsaved edits AND a drifted file is the genuine
conflict — could surface it more loudly than the per-tool flag.

** TODO Dogfooding friction (tradingsim review fixes, 2026-06) [#C]
Three from a ~30-call editing session, all minor:
- ~re-search-backward~ is void ("Variable definition is void") while
  ~search-backward~ (literal) exists — the regex-backward sibling is
  the natural reach when anchoring "the thing just above point"
  (here: a ~#[tokio::test]~ attribute above a defun).
- Rust "defun" excludes preceding ~attribute_item~s, so
  delete/replace-a-test needs a manual ~search-backward~ to the
  attribute line before ~delete-region~. An attribute-inclusive
  defun span (or a ~treesit-defun-with-attrs~ range helper) would
  make "remove/replace this test" a one-liner.
- Passing ~session: "default"~ after a path-keyed open errors with
  "no such session: default"; when other sessions are warm, the
  error could list the live session names (the path) as a hint.
Otherwise smooth: transactional ~edits~ batches (up to 6 per call)
plus the save-time tree-sitter warning carried the whole session;
zero rollbacks needed.

** TODO Span-bounded search (anchors past an explicit BOUND)
With an explicit search BOUND, ~$~/~\b~ treat the haystack cut as a
boundary where Emacs consults the real buffer (it backtracks the match
to fit the bound AND evaluates assertions against the text past it).
The high-level ~regex~ API can't express both; ~regex-automata~'s
~Input::span~ can — matches confined to a span, look-around reading
outside it. Switch the searches to it and the divergence (documented on
~Buffer::re_search_forward~) closes.

** TODO replace-match FIXEDCASE/LITERAL args
It always expands backrefs; the optional Emacs arguments are unimplemented.

** TODO forward-line shortfall at the top boundary
Landing on a line beginning that sits exactly at point-min (or buffer start)
reports a shortfall of 1 where Emacs reports 0 — the position is right, only
the return value diverges. Pre-existing, identical in both stores; fixing it
means teaching the backward scan that "no newline above, but already at a
line beginning" is a complete move.

** TODO Per-program resource limits
A wall-clock/CPU bound per program — blocked on tulisp cooperative
cancellation (eval interruption), see the wishlist below.

** TODO Vocabulary: keep-lines / flush-lines / sort-lines / case-fold [#C]
Classic Emacs bulk ops agents reach for constantly: "delete every
line matching X" is a hand-rolled loop today; ~keep-lines~ /
~flush-lines~ are one-liners over the streaming search. ~sort-lines~
(region) and a ~case-fold-search~ knob (or ~:case-insensitive~ on the
search/occur tools) round it out. Smaller parity: ~kill-whole-line~,
~forward-paragraph~, ~back-to-indentation~, ~current-indentation~.

** TODO Auto-revert before rehearse takes its snapshot [#C]
~rehearse~ skips auto-revert (the revert would land before the
rollback snapshot), so a rehearsal on a CLEAN but stale buffer
previews the edit against bytes that the committing ~run_program~
will then apply against the re-read file — preview and commit can
disagree. Auto-reverting a clean buffer first (then snapshotting)
is side-effect-free and closes the gap.

* MCP / agent UX

The agent-facing tool surface. Recurring lens: conveniences that
matter MOST for less capable models — defaults that make wrong edits
an error instead of a silent outcome, one-call recovery, and
discoverability over memorized conventions.

** TODO Uniform session addressing on every tool [#A]
~checkpoint~ / ~restore_checkpoint~ / ~list_checkpoints~ /
~save_buffer~ take only ~session~ — the path-keyed one-call flow
breaks exactly at the safety tools (after editing via ~{path}~, a
checkpoint needs the canonical-path session id). Route them through
~resolve_session~ like everything else, and rename ~save_buffer~'s
destination to ~to~: today ~path~ means "which session" on every
tool except save_buffer where it means "destination file" — one
parameter name with two meanings is a trap weaker models will hit
every time.

** TODO Auto-checkpoint each mutating call + an undo_last tool [#A]
Snapshots are O(1): capture one implicitly before every mutating
tool call (bounded ring, ~8 per session) and expose
~undo_last {path}~. Makes "oops, wrong anchor" a one-call recovery
instead of requiring checkpoint/rehearse discipline up front —
discipline less capable models won't have. Complements (does not
replace) the TUI section's granular-undo question.

** TODO expect_unique on replace_text; consider making it the default [#A]
First-match semantics silently edits the wrong site when the anchor
text also occurs earlier in the buffer. An ~expect_unique: true~
flag (per edit in ~edits~ batches too) errors with the match
positions when the pattern hits more than once — the built-in Edit
tool's one safety property mime lacks. Also report the match's line
number in the success message ("replaced at line 312") so a
misanchored edit is visible without a follow-up view. For weaker
models ambiguity-as-error is the right DEFAULT; bake the flag in,
then consider flipping it.

** TODO close_session tool + bounded warm sessions [#B]
Sessions accumulate forever, each holding an open fd (see the
paged-reader fd item) and a warm buffer; an agent can neither
release a finished file nor force a clean re-open.
~close_session {path|session, force?}~ — refuse on unsaved edits
without ~force~ — plus optional LRU eviction of clean idle sessions.

** TODO outline tool + defun-scoped edit addressing [#B]
~outline {path}~ = ~treesit-list-defuns~ as a first-class tool: the
natural first move on any code file, discoverable from the schema
instead of requiring lisp boilerplate. Then ~scope: {defun: "name"}~
on replace_text/occur (goto + narrow + edit + widen internally) so
"replace X inside function Y" needs no program at all — and fold the
older "insert relative to a defun/node" item in as an ~anchor~
parameter on insert_text. This is the path by which weaker models
get structural editing without learning the treesit vocabulary.

** TODO help tool + schema-generated docs [#B]
~help {topic}~ returning the canonical cheat-sheets (regex dialect,
treesit vocabulary, conflict workflow, a builtin's doc) lets the
always-loaded skill prompt shrink to a pointer, and lets a model
look details up instead of guessing. Builtin-level help wants the
obarray introspection on the tulisp wishlist. Pair with
~mime mcp --describe-md~ emitting the tool table, so the README and
the user's skill file embed by generation — the tool contract
currently lives in three hand-synced places and drifts.

** TODO Clamp huge diffs in run reports [#B]
The canonical bulk edit (~replace-regexp~, 200k replacements)
returns a unified diff proportional to the whole change — megabytes
straight into the agent's context. Serialize head + tail (~200
lines) around an elision line carrying the suppressed hunk/line
count; ~full_diff: true~ opts out.

** TODO search tool: direction + matched-line echo [#C]
A ~direction: backward~ option (and registering ~re-search-backward~
/ ~looking-back~ in the lisp layer — see the 2026-06 friction entry)
plus echoing the matched line occur-style (~LINE @POS: text~): today
nearly every search is followed by a view just to see what matched.

** TODO Opt-in viewport echo on mutating tools [#C]
~view: N~ on run_program/insert_text/replace_text appends the
~window~ render around point after the edit — confirms a multi-line
insert landed where intended without a second call; costs tokens
only when asked for.

** TODO Multi-file batch edits (design pass first) [#C]
~replace_text {files: [path…], edits: […]}~ — explicit path list
(no globs), one transaction + stale guard + report per file, still
roots-confined and audited. Cross-file renames are the one rule-
shaped edit where the built-in tools currently beat mime on call
count. Prefer this over exposing trusted ~find-file~ through MCP:
an explicit list stays reviewable per file.

** TODO Document relative paths in the tool schemas [#C]
~check_path~ already resolves relative paths against the cwd; the
~path~ schema description should say so — saves an absolute-path
prefix per call in single-root setups, and weaker models otherwise
copy whatever convention they saw last.

* Structural editing

** TODO Byte↔char index on Syntax [#C]
Every node accessor converts byte offsets to char positions via an
O(text) prefix scan (~char_of~), so mapping accessors over a big
query's captures is O(captures × file). A line-offset index built once
per parse (or reusing the store's tree summaries) makes it O(log).

** TODO Incremental re-parse (~InputEdit~s) [#C]
Nodes are first-class values with relational navigation now; the last
piece is feeding ~InputEdit~s from buffer mutations instead of a full
re-parse per edit. Needs edit logging in both stores (byte offsets +
row/col Points at mutation time), lazily enabled on first treesit use
so non-treesit workloads pay nothing.

** TODO More languages [#C]
JS/TS, Go, …: a ~Lang~ variant + extension row + ~defun_kinds~ row each. Fix
Python ~decorated_definition~ spans (include the decorators). Adjacent: a
formatter for code edits (~rustfmt~ via ~call-process~, trusted tier only).

* TUI cockpit (mime tui)

A terminal UI over the engine — an observer/stepper first, an editor only
if daily use pulls it there. Two entry modes sharing one shell:
~mime tui SCRIPT.tl --file F~ (step a script locally) and
~mime tui --attach [S]~ (watch a daemon session live). ratatui +
crossterm behind a ~tui~ feature flag so the core stays lean; the
single-threaded ~Rc~ engine fits a TUI event loop as-is.

** TODO Phase 0 — script stepper spike [#B]
~mime tui SCRIPT.tl --file F~: in-process trusted Workspace, three panes
— buffer viewport (rendered from read_region + point, not the gutter'd
~window~ string), program pane (top-level forms, current one
highlighted), report/diff pane (per-step unified diff + reports/log).
SPACE = eval next form, q = quit (discard unless --write).
Prerequisites:
- form-by-form eval: read ONE top-level form at a time from the source
  and eval it against the warm context. If tulisp's reader can't yield
  forms incrementally, that's a tulisp wishlist item (below).
- per-step diff: store snapshot before each form, ~unified_diff~ after
  (snapshots are O(log n) for Quire — cheap even per step).
/Exit: watch a conflict-resolution script land hunk by hunk./

** TODO Phase 1 — playback controls
Auto-play with a speed setting (delay per form), pause/resume, restart
(snapshot at program start; rollback + replay — rehearse semantics make
this free), step-backward via the same snapshots (keep the last N),
finish-and-save vs finish-and-discard. Status bar: point, narrowing,
stale flag, step n/m, buffer name/version.

** TODO Phase 2 — daemon cockpit (--attach)
Watch an AGENT edit live: poll the daemon (the accept loop is
single-threaded and serves one connection at a time, so short poll
connections, not a held-open subscription) with a new read-only
~op_render {session, pos?, lines}~ + the existing op_status; panes for
viewport, per-run diffs as they land, audit-journal tail, session list.
Read-only by design — the cockpit must never be a second writer.
Stretch: a REPL pane that sends forms to the watched session (it IS a
warm single writer, so interleaving with the agent needs a turn-taking
rule or an explicit "take control" toggle).

** TODO Phase 3 — polish
- treesit-driven syntax highlighting of the viewport (the persistent
  parse + node spans are exactly a fontifier's input).
- edit-level animation: step INSIDE a form (watch replace-regexp sweep
  its matches) — needs the store edit-event log, the same
  infrastructure incremental re-parse (~InputEdit~s) wants; build once,
  serve both.
- conflict hunks rendered with side colouring; ~conflict-context~ as a
  pane.

checkpoint mechanism is too coarse for interactive editing) and keymaps
