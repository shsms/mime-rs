*** Approach (decided 2026-06-24): in-process git sequencer on a Rust git library — do NOT shell out to ~git~

Three approaches were weighed:

1. Shell out to ~git rebase -i~ with editor shims (~GIT_SEQUENCE_EDITOR~ /
   ~GIT_EDITOR~ → a ~mime~ sub-process that hosts the todo/message file and
   blocks until the agent resumes). REJECTED. (a) The MCP server is
   stdio-only and holds sessions in-process (~HashMap<String, Workspace>~,
   mcp.rs:76); a separately-spawned shim CANNOT see the agent's warm
   sessions, so the rendezvous degrades to an on-disk scratch file + sentinel
   handshake. (b) Shelling out re-opens every side-channel that makes ~git~
   unsafe for a sandboxed agent — hooks, ~--exec~, gitattributes
   clean/smudge filters, ~core.fsmonitor~, ~ext::~/ssh transports — i.e.
   arbitrary code execution + network. (c) Breaks the "no process spawn"
   invariant (safety.rs:1-10).

2. MCP server spawns the ~git~ binary directly. REJECTED for (b)+(c) above:
   ~git rebase --exec~ and hooks run arbitrary code, so a sandboxed agent
   able to invoke ~git~ has a full sandbox escape.

3. *In-process git "sequencer" on a Rust git library (CHOSEN).* Drive
   rebase / cherry-pick / revert as a sequence of pick → 3-way-merge →
   commit steps that mime owns end to end. This dissolves the ~$EDITOR~
   problem: history manipulation (reorder / squash / reword / drop) is DATA
   the agent passes or edits, never an editor invocation. The only hard
   primitive is a 3-way merge that writes conflict markers into the worktree
   — which routes straight into the existing conflict tools.

Why a library, not the ~git~ binary:
- No process spawn → mime's "no shell, no process spawn, no network"
  invariant (safety.rs:1-10) is PRESERVED. Git becomes in-process Rust calls,
  like the existing file ops.
- No shell-out side-channels. ~git~ is dangerous in a sandbox not for its
  data ops but for hooks / ~--exec~ / clean+smudge filters / fsmonitor /
  pager / editor / ~ext::~ transports. A library has none of these unless we
  wire them — "git minus code-execution" is the DEFAULT, not something we
  fight the binary to suppress.
- Network blocked by OMISSION, not by filtering flags. fetch / push / clone /
  remote are a SEPARATE API surface; the sequencer never calls or exposes
  them. Defense in depth: reject any op needing a remote; never register
  remote/transport tools. (This is the clean form of "block push/pull/remote"
  — clean because it's a different code path, not a flag on one omnipotent
  binary.)

Security checklist — block the non-network arbitrary-code channels too (they
are subtler than the network ones):
- Hooks: never run ~.git/hooks~. (libgit2 does not run hooks itself; if
  hand-rolled, never invoke them. Bonus: deterministic, no surprise side
  effects.)
- ~--exec~ / arbitrary step commands: do not implement.
- gitattributes process filters (clean/smudge): never register process-filter
  callbacks; built-in ident/crlf are fine.
- ~core.fsmonitor~ / ~core.sshCommand~ / ~core.pager~ / ~core.editor~ /
  aliases / ~http.*~ / ~url.*.insteadOf~: moot once remote/exec/editor paths
  are unexposed, but audit that the chosen lib honors no repo-local config
  key that shells out.
- Confine to repos under MIME_ROOTS: ~safety::check_path~ the repo path,
  refuse repos outside the allowlist (mirrors open_file, mcp.rs:551).

Library: ~git2~ (libgit2 bindings) — DECIDED 2026-06-24. It has complete,
mature merge / cherry-pick / checkout-with-conflict-style TODAY, so the
DISCRIMINATING capability (3-way merge → diff3 markers in the worktree) is
proven, not in-flight — exactly the risk gix carried. Pure-Rust ~gix~ was
considered and DEFERRED: ~gix-merge~ (its conflict-marker merge) is the newer,
less-proven piece; revisit a migration once it is production-grade (the
sequencer's library surface is small, so a later swap is contained).
- C dependency: build libgit2 from source via the ~git2~ ~vendored~ feature
  (no system libgit2 needed; adds a ~cc~ toolchain + build time — fine, the
  VM can install build tools).
- KEY API NUANCE: do NOT use libgit2's high-level ~Repository::rebase~ — it
  replays the commits between upstream and branch as plain picks and does NOT
  express reorder / squash / reword / drop. HAND-ROLL the sequencer over the
  low-level primitives: ~cherrypick_commit~ / ~merge_commits~ → in-memory
  ~Index~; ~checkout_index~ with ~CheckoutBuilder::conflict_style_diff3~ to
  write markers to the worktree; ~index.write_tree~ + ~Repository::commit~ to
  land each step; ref update to move the branch; ~reset(.., Hard)~ to the

- Hooks: libgit2 does not run ~.git/hooks~ — nothing to suppress, but never

*** The sequencer

mime implements git's "sequencer" — replay a list of pick/commit steps with
conflict stops — unified across rebase, cherry-pick, and revert (git unifies
these too: .git/sequencer + CHERRY_PICK_HEAD / REBASE_HEAD).

State lives in ~.git~ (re-entrant + crash-safe, mirrors git), NOT only in the
warm session: each tool call re-opens the repo and reads the in-progress
operation from our own ~.git/mime-sequencer~ state file (the plan is
hand-rolled over git2's cherrypick primitives — see Approach — so libgit2's
own rebase state is unused). So ~rebase~ → ~continue~ → ~continue~ survives a
server restart and never desyncs.

Step loop, per planned commit:
1. 3-way merge (cherry-pick) the commit onto current HEAD;
2. clean → create the commit (planned message for reword/squash), advance,
   recurse;
3. conflicted → write markers to the worktree, RELOAD any warm buffer whose
   file changed (mime is the worktree authority — it knows exactly which
   files moved; reuse ~Quire::rebase_to~, quire.rs:852 / revert), then STOP,
   returning a conflict state.

*** Agent-facing MCP tools (new ~git_*~ group)

History ops (start an operation; each may return a conflict stop):
- ~git_rebase~ {repo, upstream | onto, plan?} — plan = list of {commit,
  action: pick|reword|squash|fixup|drop, message?}; list order = new order.
  Omit plan → pick-all in order. No ~$EDITOR~, no todo buffer; the plan is
  data. (Optional sugar: expose the plan as an editable buffer for users who
  prefer to hand-edit it — not required.)
- ~git_cherry_pick~ {repo, commits, onto?}
- ~git_revert~ {repo, commits}

Sequencer control (polymorphic over the in-progress op, like git's
sequencer — not per-op):
- ~git_continue~ {repo} — after the agent resolves conflicts with the
  existing conflict tools and saves, commit the resolved tree and advance;
  returns next state (another conflict stop, or done). Pre-check: refuse if
  ~conflict-count~ > 0 — a free safety net.
- ~git_skip~ {repo} — drop the current step, advance.
- ~git_abort~ {repo} — restore HEAD/refs/worktree to the pre-op snapshot.
- ~git_status~ {repo} — {operation, current_step, done/remaining steps,
  conflicted_files, head}. The poll/inspect call.

Read/inspect (safe; for building a plan):
- ~git_log~ {repo, range?} — commits (hash, subject, author).
- ~git_show~ {repo, commit} — metadata + diff.

Conflict resolution adds NO new surface — the agent uses the existing
~conflicts~ / ~conflict-context~ / ~conflict-keep~ / ~conflict-replace~ /
~conflict-resolve-trivial~ tools on the conflicted worktree files, saves,
then calls ~git_continue~. Naming: ~git_*~ prefix groups them; verbs mirror
git so the agent's git mental model transfers.

*** Safety (decided 2026-06-24: always-on + recovery ref)

History rewriting is higher-CONSEQUENCE than text editing (a bad reset/rebase
can orphan committed work, otherwise recoverable only via the reflog), but it
adds no new AMBIENT AUTHORITY: same MIME_ROOTS confinement (~check_path~ on the
repo path), no shell/network/hooks/exec. We were initially going to gate it
behind an opt-in env (~MIME_ENABLE_GIT~, default OFF); decided instead — sole
users — to make it ALWAYS-ON and meet the blast-radius concern with a
first-class safety net: each destructive op stamps ~refs/mime-backup/<branch>~
at the pre-op tip BEFORE touching anything, so the original state is always one
~git reset --hard~ away (not reflog archaeology). Revisit a gate / safer default
(and/or a read-only vs mutating split) if mime grows non-trusting users.

*** Interim (works today, zero mime changes)

The agent can already drive ~git rebase~ via Bash and open the conflicted
files in mime to resolve them — that is the stop-gap. The sequencer is the
productized, deterministic, ~$EDITOR~-free version that makes mime the single
worktree authority.

From driving mime to rewrite this repo's own history (2026-06): the common
task "I just made this change — fold it into the commit it belongs to" has no
one-call path. Each fold took (a) a raw ~git add~ + ~git commit~ to make a temp

every commit oid from the target onward listed in order with the fixup slotted
in. Step (b) is the friction — transcribing untouched ~pick~ steps just to say
"fixup A into B"; on a long branch it is tedious and easy to drop or reorder a
commit. ~rehearse~ (tree-IDENTICAL check) makes it safe, not quick.

1. *Autosquash-style SPARSE plans for ~git_rebase~.* Let the plan express only
   the RELATIONSHIP and auto-pick the rest of ~onto..HEAD~ in order — e.g.
   ~plan: [{action: fixup, commit: X, into: Y}]~ moves X to sit under Y as a
   fixup and picks every other commit unchanged. This is git's
   ~--autosquash~ semantics as data (no ~fixup!~-subject magic needed). Removes
   the enumerate-untouched-commits chore with no new tool.

2. *A ~git_fixup~ / absorb tool.* One rehearsable call taking a target commit +
   a source — a commit, OR the staged/worktree changes directly — that does the
   temp-commit → move-under-target → fixup dance internally and lands the
   amended history. A mime-created temp commit is fine here even though mime
   runs no hooks: ~fixup~ DISCARDS the temp message and PRESERVES the target's
   existing (already-signed) message, so no sign-off is dropped. Stretch: a
   ~git absorb~-style AUTO-TARGET that maps each worktree hunk to the commit
   that last touched those lines — i.e. the find-the-commit half tracked under

   commit introduced X" becomes one call.

Relation to existing items: the blame-shaped "which commit owns this?" read is
the DISCOVERY half (find the target); this is the ACTION half (fold into it).
The split/move item is the inverse — repartition what is ALREADY committed.
Together they are one vocabulary: find-the-commit → fold-in / split-out.

From driving mime to rebase a stacked branch (2026-07-07): branch B sits on base A; A's top commits were REORDERED (rewritten to new oids, identical patch-ids). Re-rebasing B onto the rewritten A via ~git_rebase {onto: A}~ with the DEFAULT (plan-less) pick-all replayed A's rewritten commits a second time — B ended up with duplicate commits — because the merge-base with the rewritten A had moved BELOW the rewrite, so ~onto..HEAD~ still contained B's old copies of those commits and pick-all replayed every one verbatim.

Stock ~git rebase~ drops commits already present upstream by patch-id (cherry-pick / already-applied detection — the ~--no-reapply-cherry-picks~ default), so the same "reorder the base, then rebase the stack" sequence yields NO duplicates. mime's default pick-all has no such detection, so the natural stacked-branch workflow silently duplicates; recovering needs an explicit plan with a ~drop~ on each rewritten commit (what I did to fix it).

Fix: on the plan-less rebase, omit commits in ~onto..HEAD~ whose patch-id already exists in ~onto~ (match git's cherry-pick detection), or at minimum WARN and report them so the duplication isn't silent. Consider a ~reapply_cherry_picks~ flag for the rare keep-them case. rehearse already lists the resulting commits — flagging the would-be duplicates there would catch it pre-apply too.

Done [2026-07-07] on branch ~rebase-drop-cherry-picks~: the plan-less pick-all now drops commits already present in ~onto~ by patch-id (matches git's ~--no-reapply-cherry-picks~ default) and REPORTS the skipped commits so it's never silent; a ~reapply_cherry_picks~ flag overrides.
